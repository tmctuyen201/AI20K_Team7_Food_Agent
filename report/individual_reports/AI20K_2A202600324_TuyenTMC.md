```python
def check_guardrails(state: AgentState) -> GuardrailResult:
    checkers = [
        _check_ambiguous_location,
        _check_zero_results,
        _check_max_retries,
        _check_midnight_filter,
        _check_mock_location,
    ]
    for checker in checkers:
        result = checker(state)
        if result.triggered:
            return result
    return GuardrailResult()
```
Five guardrails implemented: Ambiguous Location, Zero Result, Max Retries (≥3 rejections), Midnight Filter (22:00–05:00), and Mock Location.

**ReAct loop** (`app/agent/react_agent.py`):
```python
async def run(self, state: AgentState) -> AgentState:
    while iteration < self.max_iterations:
        messages = self._build_messages(state)
        response = await self._call_llm(messages)
        tool_calls = self._extract_tool_calls(response)
        if not tool_calls:
            state["final_response"] = self._extract_text(response)
            state["is_done"] = True
            break
        for tc in tool_calls:
            result = await self._execute_tool(tc, state)
            messages.append({"role": "assistant", "content": str(tc)})
            messages.append({"role": "user", "content": f"Tool result: {result}"})
        guardrail_result = check_guardrails(state)
        if guardrail_result.triggered:
            state["final_response"] = guardrail_result.message
            state["is_done"] = True
            break
```

## II. Debugging Case Study (10 Points)

*Analyze a specific failure event you encountered during the lab using the logging system.*

### Problem Description

During testing, the agent failed to call any tools even when the user message clearly required location data. When the user typed *"tôi đang ở đâu"* (Vietnamese for "where am I"), the agent responded with a plain text message asking for the address instead of calling `get_user_location`. The same issue occurred with *"Where am I now"* — the LLM (GPT-4o-mini via LiteLLM) returned a direct text response with no `tool_calls` in the response.

**Log source** (from `logs/session_sess_8e0fdf72.log`):
```
[info] llm_call_start  has_tools=True message_count=2
[info] llm_call_completed  elapsed_ms=3813.6 has_tool_calls=False
[info] agent_run_done  final_response_preview='Xin chào! Để giúp bạn tìm các nhà hàng...'
[warning] session_no_tools  session_id=sess_8e0fdf72
```

The log clearly shows `has_tool_calls=False` and the final response was a direct text answer rather than a tool invocation.

### Diagnosis

The root cause was **not the model** (GPT-4o-mini is capable of tool calling) and **not the tool spec** (the schema in `definitions.py` is well-formed). Investigation revealed three contributing factors:

1. **Streaming mode conflict**: The `_call_llm()` method initially passed `stream=True` for some calls. LiteLLM's tool-calling requires `stream=False` because streaming does not support reading `tool_calls` cleanly from the partial message delta. This was confirmed in `provider.py` where the code explicitly sets `stream=False` but some early test runs had streaming enabled.

2. **System prompt did not explicitly instruct the LLM to call tools**: The original system prompt was written as if the agent should "decide" rather than "must use tools." The LLM treated tools as optional suggestions.

3. **Model selection**: GPT-4o-mini sometimes prefers text responses for simple conversational queries rather than tool calls. The prompt needed to be more directive.

### Solution

Three changes were applied:

1. **Enforced `stream=False`** in both `llm_chat()` and `llm_chat_sync()` in `provider.py`. Added a log line `llm_config_resolved` to confirm the model and provider were resolved correctly.

2. **Updated system prompt** in `app/agent/prompt.py` to explicitly state:
   > *"You MUST use the available tools to gather information. Do not guess or make up data."*

3. **Added logging for `session_no_tools`** in `app/main.py` as a warning so that developers can immediately spot when the LLM skips tool calling:
   ```python
   if not tool_calls:
       logger.warning("session_no_tools", session_id=session_id)
   ```

After these fixes, the agent correctly calls `get_user_location` when the user asks about their location, as confirmed in the second session log entry.

## III. Personal Insights: Chatbot vs ReAct (10 Points)

*Reflect on the reasoning capability difference.*

### 1. Reasoning — How the `Thought` block helped the agent

The `Thought` block in the ReAct loop is a game-changer compared to a direct chatbot. In a traditional chatbot, the LLM receives the user message and immediately produces an answer — it cannot show its reasoning. In the ReAct agent:

```
[Thought] The user wants to know their location. I need to call get_user_location.
[Action] get_user_location(user_id="u01")
[Observation] lat=21.0285, lng=105.8542, city="Hà Nội - Hoàn Kiếm", source="mock_data"
```

This explicit reasoning allowed me to trace exactly what the LLM was thinking. In contrast, a chatbot would have simply answered *"Bạn đang ở Hà Nội"* with no visibility into how it determined that. The `Thought` block made debugging tractable — when the agent failed, I could see which tool it decided to call and why, rather than guessing at the LLM's internal logic.

### 2. Reliability — Cases where the Agent performed worse than the Chatbot

The ReAct agent was noticeably **worse** in the following scenarios:

- **Simple factual questions**: "Hello", "Thank you", "What's your name?" — the agent still went through the full ReAct loop, potentially calling tools unnecessarily. A chatbot would just respond directly with zero latency overhead.

- **Single-step queries**: When the user asks something that requires just one tool call and the LLM decides to skip tool calling (as in the `session_no_tools` bug), the agent takes 3+ seconds to respond (LLM call + parsing) versus a chatbot which would be instant.

- **Ambiguous user intent**: When the user says something vague like "find me food", the ReAct agent can loop indefinitely or call the wrong tools. A chatbot could ask a clarifying question with a simple text response.

- **Tool failure cascading**: If `get_user_location` fails, the agent propagates bad state to `search_google_places` and eventually to `calculate_scores`, producing unreliable results. A chatbot would gracefully skip the tool and ask for more information.

### 3. Observation — How environment feedback influenced next steps

The `Observation` step is where the ReAct loop truly shines. In the agent's `run()` method, after each tool call, the result is appended to the message history:

```python
messages.append({"role": "assistant", "content": str(tc)})
messages.append({"role": "user", "content": f"Tool result: {result}"})
```

This creates a feedback loop where the LLM sees what each tool returned before deciding the next action. For example:

1. **Thought**: "I need the user's location before searching."
2. **Action**: `get_user_location(user_id="u01")`
3. **Observation**: `"source": "mock_data"` → LLM sees the warning and decides to ask user to confirm address.
4. **Guardrail triggers**: `_check_mock_location` sees `source == "mock_data"` and returns the address confirmation prompt.

Without this observation step, the LLM would not know that it had fallen back to mock data and might have presented inaccurate information. The structured logging in `app/core/logging.py` — specifically `log_agent_step()`, `log_tool_call()`, `log_tool_result()` — made this loop traceable in the JSONL files, which was essential for debugging.

## IV. Future Improvements (5 Points)

*How would you scale this for a production-level AI agent system?*

### Scalability

- **Async tool execution queue**: In the current implementation, tool calls are executed sequentially in `for tc in tool_calls`. For a production system with many independent tools (e.g., weather + traffic + reviews), use `asyncio.gather()` to execute them in parallel, reducing total latency.

  ```python
  results = await asyncio.gather(*[self._execute_tool(tc, state) for tc in tool_calls])
  ```

- **Tool registry with dynamic discovery**: Currently, `_route_tool()` in `react_agent.py` is a manual dictionary. For a many-tool system (50+ tools), implement a plugin registry (`app/tools/registry.py`) that auto-discovers tools from a directory, enabling team members to add tools without modifying the agent core.

- **Rate limiting with token bucket**: Implement per-user rate limiting using a token bucket algorithm to prevent abuse of expensive LLM calls and Google Places API calls.

### Safety

- **Supervisor LLM audit**: Before executing tool calls, route them through a "Supervisor" LLM that verifies the tool call is safe and within scope. For example, `save_user_selection` should be audited for data integrity before writing to the history store.

- **Structured output validation**: Add a JSON schema validator between the LLM response and tool execution to catch malformed arguments before they reach the tool handler. This prevents type errors and hallucinated tool arguments.

- **Human-in-the-loop escalation**: After 3 consecutive rejections (Max Retries guardrail), instead of just asking for clarification, route the conversation to a human operator with the full session history loaded.

### Performance

- **Vector DB for tool retrieval**: In a system with 50+ tools, the LLM cannot hold all tool descriptions in context. Use a vector database (e.g., Qdrant or Weaviate) to retrieve the top-k relevant tools for each query, reducing prompt size and improving accuracy.

- **Caching layer for Google Places**: Add Redis caching for Places API responses (keyed by `lat+lng+keyword+radius`). Google Places API has rate limits; caching popular queries (e.g., "phở near Hoan Kiem") significantly reduces API costs and latency.

- **Streaming responses with WebSocket**: The current `run_streaming()` yields step-by-step events but the LLM still processes the full response. For true streaming, implement token-level streaming via FastAPI WebSocket (`app/api/chat.py`) to deliver partial responses as they are generated, improving perceived latency.
